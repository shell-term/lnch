# ロードマップ・開発計画

## 1. バージョン概要

| バージョン | 機能 | 概要 |
|-----------|------|------|
| **v0.1 (MVP)** | コア機能 | YAMLを手書き → `lnch` でTUI起動 → 一括管理 ✅ |

## 2. v0.2 — ログ検索 + 設定リロード

### ログ検索

**概要**: TUI内で vim/less ライクなログ検索機能

lnch の強みである TUI ログ閲覧を強化。プロセスが多いほどスクロールだけでは目的の行を探しにくくなるため、検索は実用上大きな価値がある。

**主な変更点:**
- `/` キーで検索バーを開く
- `n` / `N` で次/前のマッチへジャンプ
- マッチ箇所をハイライト表示
- `Esc` で検索を閉じる

### 設定リロード

**概要**: YAMLを編集後、lnch を再起動せずに反映できる機能

**主な変更点:**
- キーバインド（例: `F5`）で yaml を再読み込み
- 追加/削除されたタスクを反映
- 変更されたタスクは再起動、変更のないタスクはそのまま維持
- バリデーションエラー時はステータスバーに通知して変更を適用しない

**設計方針:**
- 手動トリガー（キーバインド）のみ。ファイル監視による自動リロードは将来のオプションとして検討
- 編集途中の不完全な yaml を読み込むリスクを避ける

## 3. v0.3 — `lnch init`

**概要**: `lnch init` で対話的にYAMLを生成

設定ファイルが無い場合のエラーメッセージで既に `lnch init` を案内しているため、期待を裏切らないためにも実装する。

**主な変更点:**
- `lnch init` サブコマンドの追加
- 対話型プロンプトでプロジェクト名・タスクを入力
- `lnch.yaml` を生成

**CLI拡張:**

```
lnch init                    # 対話的にlnch.yamlを生成
```

---

## 4. v0.4 — ログファイル出力

**概要**: タスクのログをファイルに保存

lnch を閉じるとログが消える。デバッグやチームメンバーへの共有のために、ログの永続化を提供する。

**主な変更点:**
- タスクごとのログファイル出力オプション
- グローバルオプション `--log-dir` で全タスクのログを一括保存

**設定ファイル拡張例:**

```yaml
tasks:
  - name: api
    command: cargo run
    log_file: ./logs/api.log
```

**CLI拡張:**

```
lnch --log-dir ./logs        # 全タスクのログを指定ディレクトリに保存
```

---

## 5. v0.5 — stdin パススルー

**概要**: TUI 上から子プロセスの stdin に入力を送れるようにする

現状 lnch は子プロセスの出力を表示するだけで、入力を送る手段がない。対話プロンプト（`Proceed? (y/n)` 等）を出すコマンドが入力待ちでハングする。

**主な変更点:**
- 入力モードの切り替え（vim の Normal/Insert のようなモード管理）
- 入力モード中はキー入力を選択中タスクの stdin に転送
- ConPTY（Windows）/ pipe（Unix）両方で対応
- ステータスバーにモード表示（`[INPUT]` / `[NORMAL]`）

## 6. v0.6 — 自動再起動

**概要**: プロセスクラッシュ時の自動再起動ポリシー

開発環境だけでなく、本番・ステージング環境での可用性向上にも対応する。

**主な変更点:**
- タスクごとの `restart` ポリシー設定（`never` / `on_failure` / `always`）
- 自動再起動のバックオフ戦略（指数バックオフ、最大再試行回数）
- 既存の `ready_check` との連携（再起動後の準備完了判定）

**設定ファイル拡張例:**

```yaml
tasks:
  - name: backend
    command: cargo run -- --port 8080
    restart: on_failure         # never | on_failure | always
    max_restarts: 5
```

## 未解決: プログレスバー表示の改善

**状態**: 2回試行して断念。将来再挑戦の余地あり。

**問題**: プログレスバーが `\r` で同じ行を上書きする仕組みだが、lnch の行ストリームモデルと噛み合わず、ログが崩れる。

**試行1 — `\r` 処理 + LogBuffer 行置換**:
- `process_cr()` で行内の `\r` を分割し最後のセグメントのみ残す
- `LogBuffer::push_or_replace()` で連続する上書き行を差し替え
- パイプモードでは `\r` がそのまま残るため有効
- ConPTY モードでは `\r` が `\x1b[H`（カーソルホーム）に変換され、`strip_ansi` 後にテキストが連結される問題が残った

**試行2 — ConPTY 用 `\x1b[H` 分割の追加**:
- `process_conpty_overwrite()` で `strip_ansi` 前に `\x1b[H` で分割
- テストでは正しく動作したが、実際の TUI 上での表示は改善しなかった
- ConPTY の出力フォーマットが環境やバッファリングに依存し、安定した処理が困難

**根本的な課題**: lnch は出力を「行ストリーム」として扱うが、プログレスバーは「画面上の同一位置を上書き」するターミナルモデルを前提としている。正しく対応するには VT パーサー（仮想ターミナルエミュレータ）レベルの処理が必要になる可能性がある。
